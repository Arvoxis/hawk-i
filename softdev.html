01

The Big Picture

Before writing a single line of code, you need a clear mental model of what the software is actually doing. Think of Hawk-I's software as three separate systems talking to each other:

System A — On the Drone

Edge Inference

This runs on the Jetson (the small computer attached to the drone). It captures the camera feed, runs the AI models, and sends results over WiFi to your laptop. You don't write this code — the HW team does. But you need to understand what it sends you.

System B — Your Laptop

Backend Server

This is a Python FastAPI server running on your laptop. It receives data from the drone, runs the heavier AI models (SAM 3, DINOv2, LLM), stores everything in a database, and serves the dashboard. This is your main job.

System C — Browser

Dashboard

A Streamlit web app that opens in your browser. It shows the live drone feed with AI boxes drawn on it, a GPS map with defect pins, a live report panel, and a text box where the engineer types what to search for. This is your second job.

What the drone sends you

The Jetson sends you a JSON message over WebSocket for every detection it finds. Each message contains: the defect class name (e.g. "cracked concrete"), confidence score (0 to 1), the bounding box coordinates (x1, y1, x2, y2 in pixels), the GPS location (latitude, longitude, altitude), and a JPEG-compressed image of the frame. Your backend receives this and does everything else.

02

How Each AI Model Works

There are 4 AI models involved in Hawk-I. Two run on the drone (HW team's responsibility), two run on your laptop (your responsibility). Here's what each one does in plain English.

Model 1 — YOLO11n (runs on drone, not your job but good to know)

What it is: YOLO stands for "You Only Look Once." It's a very fast object detection model that looks at a camera frame and draws boxes around things it recognises. The "11n" means it's the 11th version, "nano" size — the smallest and fastest variant.

How it works: It divides the image into a grid. For each grid cell, it asks "is there an object here, what class is it, and where exactly is its bounding box?" It does this for the whole image in one single pass — that's why it's fast enough to run at 60 frames per second.

Its limitation: It can only detect things it was trained on. If you trained it on "crack" and "rust," it cannot detect "missing bolt" unless you retrain it. This is why we add Grounding DINO on top.

runs on Jetson60+ FPSfixed classes only

Model 2 — Grounding DINO (runs on drone, not your job but very important)

What it is: Grounding DINO is a "foundation model" — trained on hundreds of millions of image-text pairs from the internet. It understands both images and language simultaneously. DINO stands for "Detection In an Open-world" — it can detect anything you describe in text.

How it works: You give it two inputs — an image AND a text description. Inside the model, a vision encoder (like a very smart image reader) and a language encoder (like BERT, which understands sentences) both process their inputs and then cross-reference each other. The model asks "where in this image does the text description best match?" and draws a bounding box there.

Why it's the magic: The engineer types "spalling . exposed rebar . rust stain" (using a dot to separate multiple queries) into your dashboard. Your backend sends this text string to the drone via WebSocket. The Jetson feeds it into Grounding DINO. The model finds every matching instance — without ever being trained on those specific words. That's the zero-shot part.

What you receive from it: A list of bounding boxes, each with a matched phrase and a confidence score. E.g. {"phrase": "rust stain", "box": [120, 340, 280, 490], "confidence": 0.71}

runs on Jetsontext-promptedzero-shot8–12 FPS

Model 3 — SAM 3 (runs on YOUR laptop — your responsibility)

What it is: SAM stands for "Segment Anything Model," made by Meta. Version 3 is the latest. Where YOLO and Grounding DINO draw rectangular bounding boxes, SAM draws pixel-perfect outlines around the exact shape of the object. This is called segmentation.

How it works: You give SAM two things — the original image and a "prompt." The prompt can be a bounding box (which is what we use — the box that Grounding DINO already found). SAM then asks "within and around this box, what are the exact pixels that belong to this object?" and returns a binary mask — a black and white image where white pixels = the defect.

Why we need it: The bounding box from Grounding DINO just gives you a rectangle. But to measure the actual crack length or the area of water damage, you need the precise shape. SAM gives you that. Once you have the mask, you count the white pixels, and using the drone's altitude and camera focal length, you convert that pixel count to real-world centimetres squared.

The maths for pixel → cm conversion: real_width_cm = (pixel_width / image_width_px) × (altitude_m × sensor_width_mm / focal_length_mm) × 100. The HW team gives you altitude and focal length — you just plug them in.

YOUR JOBruns on your GPUneeds bounding box as promptoutputs pixel mask

Model 4 — DINOv2 (runs on YOUR laptop — your responsibility)

What it is: DINOv2 is a visual feature extractor made by Meta. It doesn't detect objects or draw boxes. Instead, it reads an image and produces a 768-number summary of everything it sees — called an "embedding." Think of it as a fingerprint of the image's visual content.

How we use it: During the first inspection of a structure, we store the DINOv2 embeddings of all frames — this is the "healthy baseline." On subsequent inspections, we compute new embeddings and compare them to the baseline using cosine similarity (a mathematical measure of how similar two vectors are, from 0 to 1). If the similarity drops below 0.82, it means the structure looks significantly different from before — even if no specific defect was typed in. This is how we catch "unknown unknowns."

For the hackathon: This is a nice-to-have feature, not critical path. Build it after everything else works. It only needs to run on every 10th frame to save compute.

03

The Data Flow — What Happens to Each Frame

This is the most important thing to understand. Here is exactly what happens from the moment the drone camera captures a frame to the moment a pin appears on your dashboard map.

1
Camera captures a frame on the drone
The IMX477 camera is connected to the Jetson via a MIPI CSI-2 cable. GStreamer (a video pipeline tool) reads frames from the camera at 1080p/60fps. OpenCV then receives those frames as NumPy arrays — basically a 1080×1920×3 grid of numbers representing pixel colours. The HW team sets this up.
HW teamGStreamerOpenCV
2
Two AI models run on the frame simultaneously
YOLO11n runs on every frame at 60 FPS using TensorRT (NVIDIA's optimised inference engine). Grounding DINO runs every 5th frame at ~8 FPS with the current text query. Both run in separate Python threads simultaneously. The HW team handles this, but you need to know the output format because your backend receives it.
HW teamtwo threadsTensorRT INT8
3
Detection results are packaged into a JSON payload
The Jetson creates a JSON object containing all detections from both models, the GPS coordinates from MAVLink, and a JPEG-compressed image of the annotated frame. This JSON is the exact thing your WebSocket server will receive. Here is the exact structure you need to expect and parse:
HW team sends thisYOUR server receives this
# This is what arrives at your FastAPI WebSocket endpoint { "timestamp": 1712345678.432, "gps": { "lat": 12.9716, "lon": 77.5946, "alt_m": 12.4 }, "yolo_detections": [ {"class": "crack", "conf": 0.87, "box": [120, 340, 280, 490]} ], "gdino_detections": [ {"phrase": "cracked concrete", "conf": 0.71, "box": [118, 338, 285, 495]} ], "frame_jpeg": "<base64-encoded JPEG string>" }
4
YOUR backend receives it and runs SAM 3
Your FastAPI WebSocket endpoint receives the JSON above. You decode the JPEG image, take each bounding box, and pass it to SAM 3. SAM 3 returns a binary mask. You count the white pixels in the mask and run the pixel→cm² conversion formula. Now you have: class name, confidence, GPS coords, and the real-world defect size in cm².
YOUR JOBFastAPISAM 3pixel→cm math
5
Save to database and push to dashboard
You save the detection (class, confidence, GPS, area_cm2, image path, timestamp) into your PostGIS database. Simultaneously, you push a simplified version of the detection to the Streamlit dashboard via a separate WebSocket connection. The dashboard receives it and immediately drops a colour-coded pin on the Folium map.
YOUR JOBPostGIS INSERTWebSocket push
6
Every 30 seconds — batch to LLM for report entry
A background task in FastAPI collects all high-confidence detections from the last 30 seconds. It sends the annotated image + detection metadata to Gemma-3 (running locally via Ollama). The LLM returns a structured JSON with severity level, recommended action, urgency days, and a plain-language description. This gets added to the report panel on the dashboard.
YOUR JOBGemma-3 via OllamaLangChainBackgroundTasks
7
On mission end — generate PDF report
When the drone lands and the engineer clicks "Generate Report," your FastAPI endpoint queries all detections from PostGIS, sorts them by severity, takes the annotated images, generates a Folium map screenshot, and passes everything to a WeasyPrint HTML template. WeasyPrint converts the HTML to a PDF. The dashboard triggers a file download. Done.
YOUR JOBWeasyPrintPostGIS queryFolium screenshot
04

Backend — FastAPI + WebSocket + PostGIS

FastAPI is your central nervous system. Everything flows through it. Here's what it needs to do and how to structure it.

What FastAPI needs to handle

WebSocket endpoints

/ws/drone — receives data FROM the Jetson (detections + frames). This is where all the AI inference results arrive.

/ws/dashboard — sends data TO the Streamlit dashboard. When a new detection is processed, push a summary here so the map pin appears instantly.

REST endpoints

POST /query — receives the text query from the dashboard and forwards it to the drone via the drone WebSocket.

GET /detections — returns all stored detections from PostGIS as GeoJSON for the map.

GET /report/pdf — triggers PDF generation and returns the file.

The WebSocket receiver — the most important function you'll write

# backend/main.py from fastapi import FastAPI, WebSocket from fastapi.background import BackgroundTasks import json, base64, numpy as np, cv2 from sam3_worker import run_sam3 from database import save_detection from llm_worker import llm_queue app = FastAPI() dashboard_clients = [] # list of connected dashboard WebSockets @app.websocket("/ws/drone") async def drone_receiver(websocket: WebSocket): await websocket.accept() while True: raw = await websocket.receive_text() data = json.loads(raw) # Decode the JPEG frame back to a NumPy image array jpg_bytes = base64.b64decode(data["frame_jpeg"]) frame = cv2.imdecode(np.frombuffer(jpg_bytes, np.uint8), cv2.IMREAD_COLOR) # Merge YOLO + GDINO detections all_detections = data["yolo_detections"] + data["gdino_detections"] for det in all_detections: if det["conf"] < 0.45: continue # skip low-confidence detections # Run SAM 3 to get pixel mask and area mask, area_cm2 = run_sam3(frame, det["box"], data["gps"]["alt_m"]) # Severity mapping: L1=minor, L2=moderate, L3=critical severity = "L3" if area_cm2 > 500 else "L2" if area_cm2 > 100 else "L1" # Save to PostGIS database await save_detection( class_name=det.get("class") or det.get("phrase"), confidence=det["conf"], gps=data["gps"], area_cm2=area_cm2, severity=severity ) # Push to dashboard — this makes the pin appear on the map pin_data = {"lat": data["gps"]["lat"], "lon": data["gps"]["lon"], "class": det.get("class") or det.get("phrase"), "severity": severity, "area_cm2": round(area_cm2, 1)} for client in dashboard_clients: await client.send_json(pin_data) # Add to LLM queue for report generation llm_queue.put({"det": det, "area": area_cm2, "gps": data["gps"]})

PostGIS database — what tables you need

PostGIS is just PostgreSQL with geospatial superpowers. You only need one main table for the hackathon:

-- Run this once to set up your database CREATE EXTENSION IF NOT EXISTS postgis; CREATE TABLE detections ( id SERIAL PRIMARY KEY, class_name TEXT NOT NULL, confidence FLOAT, severity TEXT, -- 'L1', 'L2', 'L3' area_cm2 FLOAT, location GEOMETRY(Point, 4326), -- GPS point (lon, lat) altitude_m FLOAT, image_path TEXT, -- saved annotated frame on disk llm_report JSONB, -- LLM output stored as JSON detected_at TIMESTAMPTZ DEFAULT NOW() ); -- Spatial index so GPS queries are fast CREATE INDEX ON detections USING GIST(location);
Quick tip

Use asyncpg (not psycopg2) for your database driver because FastAPI is async and asyncpg is non-blocking. Your save_detection function should be async def and use await. Otherwise your whole server freezes every time you write to the database.

How live pins appear on the map

This is the trickiest part of the dashboard. Streamlit re-renders the whole page on every interaction, so you can't just "add a pin" — you need to store pins in st.session_state and re-render the Folium map with all pins each time a new detection arrives.

# Store pins across rerenders using session state if "pins" not in st.session_state: st.session_state.pins = [] # Poll your FastAPI for new detections every 2 seconds import requests, time new_detections = requests.get("http://localhost:8000/detections/latest").json() for det in new_detections: if det not in st.session_state.pins: st.session_state.pins.append(det) # Rebuild the Folium map with ALL pins each time m = folium.Map(location=[12.97, 77.59], zoom_start=18) colour_map = {"L1": "green", "L2": "orange", "L3": "red"} for pin in st.session_state.pins: folium.CircleMarker( location=[pin["lat"], pin["lon"]], radius=8, color=colour_map[pin["severity"]], popup=f"{pin['class']} — {pin['area_cm2']} cm² — {pin['severity']}" ).add_to(m) st_folium(m, width=700, height=400, key="defect_map") st.rerun() # auto-refresh every 2s using st.rerun()
Important

Use st.rerun() with a time.sleep(2) at the bottom of your script to create a polling loop. This is the simplest way to make the map "live" in Streamlit without needing a complex WebSocket client inside the browser.

LLM Pipeline — Gemma-3 + LangChain

The LLM is what turns a detection event into a human-readable maintenance report entry. Here's exactly how to set it up and use it.

Setting up Ollama + Gemma-3

# Step 1: Install Ollama on your laptop # Go to ollama.com, download and install for your OS # Step 2: Pull the Gemma-3 model (do this BEFORE the hackathon — it's 8GB) ollama pull gemma3:12b # Step 3: Run Ollama (it starts a local server at localhost:11434) ollama serve # Step 4: Test it works curl http://localhost:11434/api/generate -d '{ "model": "gemma3:12b", "prompt": "Say hello", "stream": false }'

The LangChain prompt that generates the report

# backend/llm_worker.py from langchain_community.llms import Ollama from langchain_core.output_parsers import JsonOutputParser from langchain_core.prompts import PromptTemplate from pydantic import BaseModel # Define the exact structure you want from the LLM class DefectReport(BaseModel): severity_level: str # "L1", "L2", or "L3" severity_label: str # "Minor", "Moderate", "Critical" recommended_action: str # what to do urgency_days: int # how many days before action needed description: str # 2-sentence plain English summary estimated_cost_inr: str # e.g. "₹50,000 – ₹1,20,000" llm = Ollama(model="gemma3:12b", base_url="http://localhost:11434") parser = JsonOutputParser(pydantic_object=DefectReport) prompt = PromptTemplate( template=""" You are a structural engineering assistant for Indian infrastructure inspection. A drone has detected the following defect: Defect type: {defect_class} Confidence: {confidence}% Measured area: {area_cm2} cm² GPS location: {lat}, {lon} Structure type: Bridge / Public Building Based on IRC 22-2015 standards, provide a structured assessment. Respond ONLY with a valid JSON object matching this schema: {format_instructions} """, input_variables=["defect_class", "confidence", "area_cm2", "lat", "lon"], partial_variables={"format_instructions": parser.get_format_instructions()} ) chain = prompt | llm | parser # Call this function for each detection async def generate_report_entry(detection: dict) -> DefectReport: return await chain.ainvoke({ "defect_class": detection["class"], "confidence": round(detection["conf"] * 100, 1), "area_cm2": detection["area_cm2"], "lat": detection["gps"]["lat"], "lon": detection["gps"]["lon"] })
Key insight

The JsonOutputParser with a Pydantic model tells LangChain to validate that Gemma-3's output matches the schema. If the LLM returns badly formatted JSON, LangChain throws an error you can catch. Always wrap LLM calls in try/except during the hackathon — LLMs occasionally output garbage.

Your Build Order — What to Do First

The rule

Build in layers. Never build in parallel before the core works. The demo-winning moment is: text typed → drone finds it → pin appears on map. Build that first. Add every other layer on top only after that works end-to-end.

Layer 1 — The skeleton (do this first, Day 1)

Set up FastAPI with a dummy WebSocket endpoint. Make it accept a connection and print received messages. Test with a Python WebSocket client sending fake JSON. Don't touch any AI yet.
CRITICAL
Set up PostGIS with Docker. Run docker-compose up with a postgres+postgis image. Create the detections table. Test a manual INSERT and SELECT.
CRITICAL
Build the Streamlit dashboard skeleton. Just the two-column layout — left column has a placeholder image, right column has a blank Folium map centred on your test location. Text query input at the top. Nothing dynamic yet.
CRITICAL
Wire the fake detection flow. Write a script that sends a fake detection JSON to your WebSocket every 3 seconds. Make your backend receive it, INSERT into PostGIS, and serve it from GET /detections/latest. Make the dashboard poll that endpoint and drop a pin. When this works with fake data, your core loop is done.
CRITICAL

Layer 2 — Real AI (Day 1 evening / Day 2)

Swap fake data for real drone data. Have the HW team connect their Jetson WebSocket to your server. Replace your fake JSON with real detection JSON from the Jetson. Verify the same pin-drop flow works with real data.
CRITICAL
Add SAM 3. Install segment-anything-3. Write sam3_worker.py — accepts a frame + bounding box, returns (mask, area_cm2). Plug it into the WebSocket receiver after you decode the frame. Test on a saved image before connecting live.
HIGH
Add the MJPEG video feed. Write a /video_feed endpoint in FastAPI that streams the latest annotated frame as MJPEG. Embed it in the Streamlit dashboard with a raw HTML img tag. This makes the demo feel live and impressive.
HIGH
Set up Ollama + Gemma-3 and test the LLM prompt. Pull the model, run Ollama, test your LangChain chain with a hardcoded detection dict. Make sure the JSON output parses correctly via Pydantic before wiring it into the live flow.
HIGH

Layer 3 — Polish (Day 3)

Wire LLM to live detections. Use FastAPI's BackgroundTasks to call generate_report_entry() for each high-confidence detection without blocking the WebSocket loop. Push report entries to the dashboard report panel.
HIGH
Build the PDF report generator. Write an HTML template, query PostGIS for all detections sorted by severity, pass to WeasyPrint. Add a download button in Streamlit. Test with your saved detections from Day 2's flights.
HIGH
Add offline demo mode. Add a toggle in Streamlit that loads detections from a saved JSON file instead of polling the live backend. This is your safety net if the drone can't fly during the demo.
MEDIUM
Add DINOv2 anomaly scoring. Only build this if Layers 1–3 are solid. It's impressive but not critical for the demo moment.
MEDIUM
